# E006: Focused Edge Flipping (two runs, 8+8 split fleet)

**Date:** 2026-03-17 to 2026-03-18
**Target:** R(5,5) n=25
**Strategy:** tree2
**Config:** beam=80, depth=12, bias=0.8
**Init:** leaderboard
**Workers:** 16 (8 focused + 8 unfocused)
**Server:** localhost:3001 (leaderboard capacity=2000, pre-filled from E005)
**Signed:** yes (key 35a14cd3c6b2febb)

## Hypothesis

Focused edge flipping (only mutating edges in violations, per Exoo-Tatarevic
2015 Algorithm 2) would produce higher admits/hr than unfocused (all 300 edges).

## Run 1: Per-parent guilty_edges (10.6h)

**Commit:** 39b1416 (guilty_edges called per beam parent per depth)

| Metric | Focused | Unfocused | Ratio |
|--------|---------|-----------|-------|
| Rounds | 34,228 | 88,671 | 0.39x |
| Discoveries | 1,192M | 1,176M | ~1.0x |
| Admitted | 801 | 908 | 0.88x |
| **Admits/hr** | **76** | **86** | **0.88x** |
| Rounds/hr | 3,232 | 8,373 | 0.39x |
| Admit rate | 97.8% | 99.5% | ~same |

**Problem:** `guilty_edges()` called per beam parent per depth = ~960 full
clique enumerations per round. Massive overhead.

## Run 2: Once-per-depth guilty_edges (7.4h)

**Commit:** 39b1416 (guilty_edges called once per depth from best beam entry)

| Metric | Focused | Unfocused | Ratio |
|--------|---------|-----------|-------|
| Rounds | 26,873 | 62,244 | 0.43x |
| Discoveries | 836M | 825M | ~1.0x |
| Admitted | 395 | 438 | 0.90x |
| **Admits/hr** | **53** | **59** | **0.90x** |
| Rounds/hr | 3,623 | 8,392 | 0.43x |
| Admit rate | 98.8% | 99.3% | ~same |

**Still lost.** The 80x reduction in guilty_edges calls didn't help because
the overhead was never the main issue.

## Root Cause Analysis

The real problem is structural, not overhead:

**Focused mode is 2.4x more effective per iteration** (0.31 discoveries per
iteration vs 0.13 for unfocused) — targeted mutations do help find valid
graphs faster. But focused is also **2.3x slower per round on wall-clock**.

Why: With only ~40 guilty edges vs 300 per parent, focused evaluates
`40 * 80 = 3,200` candidates per depth vs `300 * 80 = 24,000`. Both are
capped at `max_iters=100,000`. Unfocused exhausts its budget in ~4 depth
levels; focused goes all 12. The extra 8 depth levels means 8 additional
beam materializations (clone graph + rebuild NeighborSet for 80 entries each).

Each beam materialization costs `O(beam_width * n^2)`. At 80 entries, 25
vertices: that's ~50,000 operations per depth. 12 depths vs 4 = 400K extra
operations per round.

The net result: focused finds 2.4x more valid graphs per iteration but
uses 2.5x more wall-clock per round, for an identical throughput. Neither
approach is better — they are Pareto-equivalent on this problem.

## Why Focused Doesn't Help Here

1. **Seeds are already near-valid.** With `--init leaderboard` and bias=0.8,
   seeds start with ~0-3 violations. Even random edge flips reach 0 violations
   quickly. The focused edge advantage (targeting ~40 relevant edges vs 300
   random) is wasted when you're already 1-2 flips from valid.

2. **Beam search width matters more than depth.** Unfocused explores 300
   edges x 80 parents per depth = 24K candidates and quickly finds the best
   single-flip mutations. Focused explores only 3.2K candidates per depth
   but goes deeper — depth doesn't help when the landscape is nearly flat.

3. **The 96-99% admission rate means quality is not the bottleneck.** Almost
   everything submitted gets admitted. The problem is discovery diversity,
   not discovery quality. Focused edges don't improve diversity.

## Verdict

**Negative result. Focused edge flipping does not help beam search with
near-valid leaderboard seeds.** The code is kept (behind `--focused false`
default) in case it helps for harder targets or different init modes.

Focused edge flipping may still help:
- With `--init random` where seeds have many violations
- At larger n (n=30+) where the valid-graph landscape is harder
- In SA/evo strategies that maintain a single graph (Exoo-Tatarevic's context)

## Decision

Default `focused` to `false`. Move to Priority 2: circulant enumeration.
