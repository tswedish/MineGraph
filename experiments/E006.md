## Analysis

### Focused was 12% slower at admits/hr

The focused approach **lost** (0.88x admits/hr). The root cause was implementation
overhead: `guilty_edges()` was called **per beam parent per depth level**, meaning
~960 full clique enumerations per round (80 parents x 12 depths). Each call
enumerates all 5-cliques and 5-independent-sets in the graph and complement.

Even though focused mode tried only ~20-50 edges vs 300, the overhead of computing
which edges to try consumed more time than it saved.

### Discovery rates were identical

Both groups found ~1.18-1.19 billion valid graphs — the same total discovery
throughput. This means focused mode wasn't finding *better* graphs, just
spending more overhead to find the same ones.

### Rounds/hr tells the story

Focused: 3,232 rounds/hr. Unfocused: 8,373 rounds/hr. A 2.6x slowdown.
The focused workers were spending most of their time enumerating cliques
to compute guilty edges, not actually searching.

## Verdict

**Focused edge flipping is a valid idea but the implementation was wrong for
beam search.** The per-parent clique enumeration is too expensive.

## Fix Applied (post-E006)

Changed to compute guilty edges **once per depth level** from the best beam
entry, then share that edge set across all 80 parents. This reduces clique
enumeration from ~960/round to ~12/round (80x reduction in overhead).

The fix should make focused mode nearly as fast per-round as unfocused (the
guilty_edges call is now ~1% of round time instead of ~60%), while still
getting the benefit of targeted mutations.

**Next experiment (E006b) should re-test with the fixed implementation.**

## Key Insight

Exoo-Tatarevic designed focused edges for single-graph steepest descent / SA,
where you compute the guilty set once every ~100 iterations on ONE graph. In
beam search with 80 parents, the cost model is completely different. The fix
is to compute once per depth level and share across parents — the guilty sets
are similar anyway since all parents are close neighbors in graph space.
